# ward

> 自宅インフラを、自然言語で・安全に、Claude Code から操作する。

_English: [README.md](./README.md)_

**ward** は、AI（[Claude Code](https://claude.com/claude-code) 上の Claude）に自宅インフラを
いじらせるための仕組みです。ディスクを見る、コンテナを並べる、イメージを pull する — そういう操作を
[MCP](https://modelcontextprotocol.io) ツールとして渡します。

肝心なのは、AI とマシンを繋ぐことじゃありません。**どこまで任せるか、事故をどう抑えるか**です。
ward はそこを中心に作っています。どの操作も必ずガードレールを通りますし、自律は一段ずつしか解放しません。

## 名前について — なぜ _ward_ か

`ward` という語が、そのままプロジェクトを言い当てています。

- **動詞なら「防ぐ」。** 事故を _ward off_ する。ガードレールで被害を封じ込める。
- **名詞なら「被後見人」。** AI を最初は監督下に置いて、信頼に応じて自律を渡していく。

頭文字は **W**atched **A**utonomy, **R**eversible **D**eployment（監督付きの自律 × 巻き戻せる変更）。

## 安全モデル

ward は既定で読み取り専用。エージェントを信用する代わりに、構造で守ります。

- **許可リストがすべて。** ward が打てるコマンドは [`operations.yaml`](./operations.yaml) に固定で
  並んでいて、`risk`（リスク）が付いています。ここにないものは実行できません。
- **必ずゲートを通る。** どの呼び出しも、実行前に `autonomy（自律レベル） × risk` で判定します。

  | autonomy \ risk | read-only | mutating   |
  | --------------- | --------- | ---------- |
  | `read-only`     | 許可      | 禁止       |
  | `approval`      | 許可      | 承認が必要 |

- **既定は読み取りだけ。** `read-only` では mutating な操作はツールにすら出てきません。
- **書き込みは提案 → 承認。** `approval` でも、mutating は頼んだだけでは動かず、いったん提案になります。
  別途承認して初めて実行 — 人間が必ず間に入ります。
- **コマンドは固定・シェルなし。** 引数列をそのまま実行し、モデルは中身を一切組み立てません。
  注入する隙がありません。
- **全部ログに残る。** 提案も実行も記録するので、後から追えて巻き戻せます。

## 自律ラダー

許す範囲は一段ずつ広げます。上げるのは設定であって、作り直しではありません。

1. **`read-only`** _(既定)_ — 見るだけ。
2. **`approval`** — 提案できる。実行は人間の承認後。
3. **`autonomous`** — ポリシー内なら承認なしで動く。**未実装。**

## 現在地

まだ初期段階です。ここまで:

- **M1** — MCP 経由の読み取り専用ステータス
- **M2** — 操作を YAML レジストリで宣言
- **M3** — 承認ゲートと、初の mutating な操作
- **i18n** — 文言の二言語化（既定は英語、`WARD_LANG=ja` で日本語）

**インストール手順はこれから。** 落ち着き次第そろえます。計画は [issues](https://github.com/ruribou/ward/issues) に。
