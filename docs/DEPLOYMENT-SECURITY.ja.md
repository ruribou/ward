# デプロイのセキュリティ

_English: [DEPLOYMENT-SECURITY.md](./DEPLOYMENT-SECURITY.md)_

ward のガードレールの強さは、その土台にある前提と同じだけしかありません——**AI から
インフラへの経路が ward だけ**、という前提です。このガイドは、その保証の「デプロイ側の半分」を
扱います。[README](../README.ja.md) はプロセス内のガードレール（ゲート・許可リスト・out-of-band
承認）を説明しますが、このガイドは、そのガードレールを迂回されないために**あなたが**ward を
デプロイするとき何をすべきかを説明します。まず README の _安全モデル_ を読んでください。

## たった 1 つのルール

**AI に substrate の資格情報を渡さない。ward をインフラへの唯一の扉にする。**

以下はすべてこのルールの帰結です。ward はどの呼び出しも `autonomy × risk` で判定し、書き込みは
人間の承認のために staged します——が、その判定が意味を持つのは、ward が substrate への
**唯一の**道の上にいるときだけです。AI の実行環境が別の経路でマシンに届けるなら、AI はその別の道を
通るだけで、ガードレールは一度も走りません。

## 2 つの正直な但し書き

ward の構造（許可リスト・ゲート・承認ツールが無いこと——README 参照）は、AI が**流れで自己承認して
しまう**ケースを塞ぎます。しかしそれ単体では、**デプロイの仕方次第**の次の 2 経路は塞げません。

1. **AI が substrate の資格情報に到達できる。** ward は操作を SSH 経由で実行します
   （`src/substrate/executor.ts`）。その SSH 鍵と設定が **AI の実行環境と同じマシン**にあると、AI は
   自分の shell から `ssh <host> …` を直接打てます——許可リストも、ゲートも、監査ログも素通りで。
   ward はもう唯一の扉ではありません。

2. **承認が AI の手の届く所にある。** 承認は人間用 CLI `ward approve <id>` です。提案ストアは
   ファイルなので、提案者と承認者は別**プロセス**です——が、その CLI が **AI と同じマシン・同じ shell**
   から届くなら、AI は自分で `ward approve <id>` を打て、self-approval に逆戻りします。
   （初期の実機検証でこれが成立したのは人間が打ったからにすぎません。それは運用上の作法であって、
   構造的な障壁ではありません。）

どちらか 1 つでも開いていれば、「人間が承認した」は迂回可能で、保証は条件付きになります。
これらを塞ぐのは、コードだけでなくデプロイの判断です。

## それらを塞ぐデプロイ姿勢

各但し書きは、AI が到達できない場所に置くことで塞ぎます。構造的な修正（資格情報の分離・承認の別
trust domain 化）は [#31](https://github.com/ruribou/ward/issues/31) で今後の作業として追跡されています。
それが入るまで、これらの姿勢が保証を保つ手段です。

| 但し書き             | やること                                                                                                                                                                                           | なぜ効くか                                                                      |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| substrate の資格情報 | SSH / cloud の資格情報を **AI の実行環境の外**に置く。ward と、それが使う鍵を、AI の Bash ツールが読めも呼べもしない場所で動かす（別プロセス・別ホスト・別アカウント、AI が持たない IAM ロール）。 | AI に `ssh` する鍵が無いので、substrate への唯一の道が ward への提案になる。    |
| 承認                 | **out-of-band** で承認する。AI が到達できない trust domain——別マシン、または push-to-approve チャネル——で行い、AI が動かす同じ shell では行わない。                                                | AI が承認ステップを実行できないので、提案者と承認者が本当に分かれたままになる。 |
| その他すべて         | **ward を唯一の経路**として扱う。ward を通さず substrate に届くサイドチャネル（2 つ目の SSH エイリアス、マウントした鍵、共有 shell）を足さない。                                                   | 迂回路のあるガードレールはガードレールではない。                                |

今日とれる最小限の安全な姿勢：ward が使う SSH 鍵が、AI の動く環境に存在せず（かつそこから呼べず）、
承認は AI がアクセスできないデバイスで人間が行う。これで構造的な作業を待たずに両方の扉を閉じられます。

## 今 ward が保証すること vs. デプロイ次第のこと

**ward の構造が保証すること（デプロイによらず）:**

- ward を通して到達できるのは [`operations.yaml`](../operations.yaml) に宣言された操作だけ。
  許可リストの外には何も無い。
- mutating な操作は頼んだだけでは動かない。_提案として staged_ され、`ward approve <id>` が消費した
  ときだけ走る——AI の MCP 面には承認ツールが無いので、ward を通して自分の提案を自分で承認できない
  （propose→approve の分離は [#28](https://github.com/ruribou/ward/issues/28) で実装）。
- コマンドは定数の引数列でシェルなしで走り、モデルはコマンドの一部すら組み立てない。提案・実行・否認は
  すべて監査ログに残る。

**あなたのデプロイ次第（まだコードでは保証されない）:**

- AI が substrate の資格情報を直接使って **ward をまるごと迂回できない**こと（但し書き 1）。
- 承認が AI の到達できない trust domain にあるおかげで、AI が **承認ステップを自分で実行できない**こと
  （但し書き 2）。

要するに：ward は propose→approve ゲートを**自分の面の中では構造的に**します。ward を**唯一の**面に
するのは、上のデプロイ姿勢——と、[#31](https://github.com/ruribou/ward/issues/31) に残る構造的な層です。

## これは単一のホストを超えて一般化する

ward は単一の ホストに SSH で始まりますが、分離の要件は provider 非依存です。substrate が AWS / GCP /
Azure になると、「AI に資格情報を渡さない」は「AI に cloud 鍵 / IAM ロールを渡さない」に変わり、
「out-of-band で承認する」はそのままです。核は、AI が回り込めない突破不能な承認・ガードレール plane——
その下の substrate は変わっても、ルールは変わりません。
