# ward

> 自宅インフラを、自然言語で・安全に、Claude Code から操作する。

_English: [README.md](../README.md)_

**ward** は、AI（[Claude Code](https://claude.com/claude-code) 上の Claude）に自宅インフラの操作を
任せられる仕組みです。ディスクを見たり、コンテナを確認したり、イメージを pull したり — そういった操作を
[MCP](https://modelcontextprotocol.io) ツールとして渡せます。

といっても、ただ AI とマシンを繋ぐだけではありません。どこまで任せるか、もし何かあっても
どう被害を抑えるか — ward がいちばん大事にしているのはそこです。だから操作はいつもガードレールを
通りますし、自律も一気にではなく少しずつ広げていきます。

## 名前について — なぜ _ward_ か

`ward` という語が、そのままプロジェクトを言い当てています。

- **動詞なら「防ぐ」。** 事故を _ward off_ する。ガードレールで被害を封じ込める。
- **名詞なら「被後見人」。** AI を最初は監督下に置いて、信頼に応じて自律を渡していく。

頭文字は **W**atched **A**utonomy, **R**eversible **D**eployment（監督付きの自律 × 巻き戻せる変更）。

## 安全モデル

ward は既定で読み取り専用。エージェントを信用する代わりに、構造で守ります。

- **許可リストがすべて。** ward が打てるコマンドは [`operations.yaml`](../operations.yaml) に固定で
  並んでいて、`risk`（リスク）が付いています。ここにないものは実行できません。
- **必ずゲートを通る。** どの呼び出しも、実行前に `autonomy（自律レベル） × risk` で判定します。

  | autonomy \ risk | read-only | mutating   |
  | --------------- | --------- | ---------- |
  | `read-only`     | 許可      | 禁止       |
  | `approval`      | 許可      | 承認が必要 |

- **既定は読み取りだけ。** `read-only` では mutating な操作はツールにすら出てきません。
- **書き込みの承認は out-of-band。** `approval` でも、mutating は頼んだだけでは動かず、いったん提案になります。
  実行できるのは、人間が別ツール `ward approve <id>` を**自分のターミナルで**叩いたときだけ。エージェントの MCP 側に
  承認ツールは無いので、**自分の提案を自分で承認できません**——承認はエージェントが触れないプロセスで起きます。
  （エージェントに substrate の鍵を直接持たせない限り成立。「扉を ward だけにする」のは次の層——
  [docs/DEPLOYMENT-SECURITY.ja.md](./DEPLOYMENT-SECURITY.ja.md) を参照。）
- **提案には plan が付く。** 提案はコマンドだけでなく、「何が変わるか」の平易な説明と、先に現状を確かめられる
  read-only コマンドを併せて示します。承認を当てずっぽうでなく _informed_ にするため。plan の生成では ホストに一切触れません。
- **コマンドは固定・シェルなし。** 引数列をそのまま実行し、モデルは中身を一切組み立てません。
  注入する隙がありません。
- **全部ログに残る。** 提案も実行も記録するので、後から追えて巻き戻せます。

## 自律ラダー

許す範囲は一段ずつ広げます。上げるのは設定であって、作り直しではありません。

1. **`read-only`** _(既定)_ — 見るだけ。
2. **`approval`** — 提案できる。実行は人間の承認後。
3. **`autonomous`** — ポリシー内なら承認なしで動く。**未実装。**

## 設定

ユーザ設定は設定ファイル（`~/.ward/config.yaml`）に置き、環境変数をプロセス単位の上書きとして使えます。
優先順位は **env > ファイル > 既定値**。単発の実行では env が常に勝ち、ファイルは MCP サーバと
`ward` CLI が共有する永続的な設定になります。

設定できるキーは 2 つ:

- **`language`** — UI 言語（既定 `en`、または `ja`）。env での上書きは `WARD_LANG`。
- **`ssh_host`** — SSH ホストの**別名**（env での上書きは `WARD_SSH_HOST`）。

```yaml
# ~/.ward/config.yaml
language: ja
ssh_host: nuc
```

CLI から設定すると、ファイルへの書き込みも行われます。

```sh
ward config set language ja
ward config set ssh_host nuc
ward config get           # 保存値とファイルパスを表示
ward config path          # 解決済みの設定ファイルパスを表示
```

> **`ssh_host` は別名のみ——実 IP やホスト名は書かない。** ward は `~/.ssh/config` 経由で解決し、
> 実アドレスはそちらに残ります。`~/.ward/config.yaml` には別名だけを書きます。ward 側でも強制しており、
> スペース・コロン・スラッシュを含む値は拒否されます。

`autonomy` は意図的にファイルでは設定できません——ガードレール属性なので `WARD_AUTONOMY` でのみ設定し、
緩めるのは常に明示的な操作であって、dotfile の静かな編集にはしません。

## 現在地

まだ初期段階です。ここまで:

- **M1** — MCP 経由の読み取り専用ステータス
- **M2** — 操作を YAML レジストリで宣言
- **M3** — 承認ゲートと、初の mutating な操作
- **i18n** — 文言の二言語化（既定は英語）。設定ファイル（`language: ja`）または `WARD_LANG` で切り替え

**インストール手順はこれから。** 落ち着き次第そろえます。計画は [issues](https://github.com/ruribou/ward/issues) に。
